# WhatsApp финансовый бот

Читает сообщения из WhatsApp-группы, парсит суммы по людям/картам/наличным
и по команде `/отчет` генерирует PDF со сводкой.

## Статус кода

- **`internal/parser`** — полностью написан и **протестирован юнит-тестами**
  на твоих реальных сообщениях (5 тестов, все проходят). Это самая важная
  и рискованная часть логики, поэтому именно её я проверил в первую очередь.
- **`internal/db`, `internal/report`, `internal/bot`, `cmd/bot`** — написаны
  по документированным API библиотек (pgx, gofpdf, whatsmeow), но **не собраны
  в этой песочнице** — сеть здесь разрешает не все домены, нужные Go-модулям
  (`golang.org`, `gopkg.in` заблокированы). На твоём сервере с обычным интернетом
  `go mod tidy` подтянет всё без проблем.

**Что стоит сделать перед первым продакшн-запуском:** прогнать `go build ./...`
и `go vet ./...` на своей машине — если API какой-то библиотеки чуть разошёлся
с версией, зафиксированной в `go.mod`, компилятор сразу укажет на конкретную
строку, это будет быстрая правка.

## Установка на сервере

### 1. Клонирование и настройка

```bash
git clone <адрес твоего приватного репозитория> whatsapp-bot
cd whatsapp-bot
cp .env.example .env
nano .env   # заполни пароль Postgres; группы настраивать не обязательно — см. раздел "Группы" ниже
```

### 2. Запуск

```bash
docker compose up -d --build     # на старых серверах: docker-compose
docker compose logs -f bot
```

При первом запуске в логах появится **QR-код прямо в терминале** — отсканируй
его с телефона бота: **WhatsApp → Настройки → Связанные устройства →
Привязать устройство**. Если QR-картинка не читается с экрана, под ней
печатается текстовый код — его можно вставить в любой онлайн-генератор QR.

**Важно:** используй отдельный номер, не свой основной — это неофициальный
протокол, есть риск ограничений на аккаунте при подозрительной активности.

После сканирования сессия сохраняется в volume `bot_data` — при рестартах
контейнера повторный QR не нужен.

### 3. Группы

По умолчанию бот учитывает **все группы**, в которых состоит его номер —
ничего дополнительно настраивать не нужно, просто добавь номер бота в каждую
группу со сборами. Учёт по всем группам общий: суммы по одинаковым именам
складываются вместе независимо от того, в какой группе пришло сообщение;
команда `/отчет` работает в любой из этих групп.

Если нужно ограничить бота конкретными группами (например, номер добавлен
и в другие, нерабочие чаты) — впиши их JID через запятую в `.env`:

```
WHATSAPP_GROUP_JIDS=120363000000000001@g.us,120363000000000002@g.us
```

Чтобы узнать JID группы: напиши в неё любое сообщение и посмотри логи —
временно можно залогировать `msg.Info.Chat` для всех чатов, либо использовать
метод `client.GetJoinedGroups()`.

### 4. Обновление бота

```bash
cd whatsapp-bot
git pull
docker compose up -d --build
```

### 5. Команды в группе

- `/отчет`, `/отчёт`, `/report`, `/итог` — бот присылает PDF за текущий месяц
  прямо в группу

## Личный ассистент (через OpenRouter)

Если написать что-то в личку номеру бота (не в рабочую группу), бот отвечает
через модель Claude, подключённую через [OpenRouter](https://openrouter.ai)
(OpenAI-совместимый API). Ассистент:

- знает текущую сводку по сбору за месяц (по всем группам сразу) и может
  отвечать на вопросы вроде "сколько уже собрали с Ахмеда";
- умеет строить отчёт за **любой период по запросу** — например, "какой сбор
  был 3 и 4 июля", "скинь отчёт за прошлую неделю", "пришли пдф за июнь".
  Он сам переводит даты в нужный формат и решает, ответить цифрами прямо
  в переписке или прислать PDF-документом — в зависимости от того, как
  сформулирован запрос (если явно просишь "документ"/"пдф"/"отчёт" — пришлёт
  файл, если просто спрашиваешь сумму — ответит текстом);
- умеет фильтровать отчёты **по конкретной группе**: "скинь отчёт по группе
  оплата клиентов за июль" — бот знает названия всех своих групп;
- считает **сбор по сотрудникам**: каждый чек привязан к тому, кто его
  отправил. "Проверь, кто сколько чеков скинул в группе сб оплата клиентов
  за июль" или "сколько чеков и на какую сумму сделал Расул" — покажет
  по имени, а если имя в WhatsApp не сохранено — по номеру телефона;
- поддерживает **дозагрузку пропущенных чеков**: если бота добавили в группу
  с опозданием (например, 3 июля, а чеки шли с 1-го), пришли пропущенные
  чеки боту в личку и напиши "запомни их, это для группы оплата клиентов" —
  он распознает каждый, проверит на дубли и сохранит в учёт с **датой с самого
  чека** (включая год), так что они лягут на свои реальные дни.

История переписки хранится в памяти процесса (не в БД) и сбрасывается при
перезапуске бота.

Чтобы включить: пропиши в `.env`:

```
OPENROUTER_API_KEY=sk-or-v1-твой_ключ
OPENROUTER_MODEL=~anthropic/claude-sonnet-latest   # можно не задавать — это значение по умолчанию
```

Ключ берётся на https://openrouter.ai/keys. Без `OPENROUTER_API_KEY` личные
сообщения просто игнорируются — вся остальная логика (учёт по группам,
отчёты, OCR) работает как обычно.

Чтобы включить: пропиши в `.env` / `docker-compose.yml`:

```yaml
ANTHROPIC_API_KEY: "твой_ключ_с_console.anthropic.com"
```

Без этой переменной личные сообщения просто игнорируются — вся остальная
логика (учёт по группе, отчёты, OCR) работает как обычно.

## OCR фото чеков

Бот умеет распознавать текст на фото чеков двумя способами (переключается
переменной `OCR_PROVIDER` в `docker-compose.yml`):

### Вариант 1 — Tesseract (бесплатно, по умолчанию)

Ничего покупать не нужно, движок уже встроен в Docker-образ. Точность
на кривых/тёмных фото заметно ниже, чем у облачных сервисов — используй
для старта, посмотри, устраивает ли качество распознавания.

### Вариант 2 — Yandex Vision (платно, точнее)

**Где купить/подключить:**

1. Зайди на **https://cloud.yandex.ru**, зарегистрируйся (можно через аккаунт Яндекса)
2. В консоли создай **платёжный аккаунт** — привяжи карту. Новым пользователям
   обычно дают небольшой грант на пробу (сумма и условия меняются, смотри
   актуальные условия на сайте при регистрации)
3. Создай **каталог** (folder) — это рабочее пространство проекта
4. В разделе **IAM → Сервисные аккаунты** создай сервисный аккаунт,
   назначь роль **`ai.vision.user`**
5. Для этого сервисного аккаунта создай **API-ключ** (не путать с IAM-токеном —
   он не истекает, удобнее для сервера)
6. Скопируй значения:
   - `YANDEX_OCR_API_KEY` — сам ключ
   - `YANDEX_FOLDER_ID` — id каталога из шага 3

**Стоимость:** тарификация Yandex Vision — за количество распознанных страниц/изображений,
цену на момент подключения смотри на странице тарифов Vision в Yandex Cloud
(она может меняться, поэтому точную цифру называть не буду — проверь при регистрации).
Для чеков одной кофейни в месяц расход обычно совсем небольшой — по несколько
рублей за десятки фото.

Чтобы включить, в `docker-compose.yml` раскомментируй три строки в `environment`
и пропиши свои значения:

```yaml
OCR_PROVIDER: "yandex"
YANDEX_OCR_API_KEY: "твой_api_ключ"
YANDEX_FOLDER_ID: "твой_folder_id"
```

## Дубли чеков

Если один и тот же чек прислали дважды (случайно переслали фото повторно),
бот это замечает и предупреждает в чат: `⚠️ Похоже, этот чек уже
присылали: Имя, 30 000 ₽, чек от 02.07.2026 14:12`. Второй раз сумма
**не учитывается** в отчётах (но сам чек сохраняется в базе для истории).

Проверка идёт в два шага:
1. Если на чеке есть номер документа/код авторизации — сверяет их напрямую
   (это ID самой банковской операции, самый надёжный способ).
2. Иначе — сверяет получателя, сумму и **время самой операции с чека**
   (не время пересылки в WhatsApp) с окном ±5 минут.

Для этого бот теперь распознаёт дату и время прямо с фото чека (если банк
её печатает — так делают большинство: "26 июня 2026 22:58:35", "Дата
операции 27.06.2026 11:43:33" и т.п.). Это же время используется в отчётах:
если попросить у ассистента "скинь чеки за 2 июля", он покажет именно то,
что реально прошло по чекам 2 июля, а не то, что переслали в чат в этот день.

## Известные ограничения парсера (на что обратить внимание)

- Новые варианты написания имён (кроме уже прописанных Пияна/Пиян,
  Хадижат/Хадижа) нужно добавлять в `contact_aliases` в БД или в
  `internal/parser/aliases.go`
- Фото сохраняются в `data/receipts/` — можно периодически проверять глазами,
  особенно первые недели, пока не станет ясно, насколько хорошо OCR
  справляется именно с вашими чеками
- Если человек напишет сообщение в формате, которого нет в тестах, строка
  попадёт в лог "Не распознано" — стоит периодически проверять логи бота
  первые недели использования и дополнять парсер

## Как добавить новый алиас имени вручную

```sql
INSERT INTO contact_aliases (contact_id, alias)
SELECT id, 'новый_вариант_написания' FROM contacts WHERE canonical_name = 'Каноничное Имя';
```
